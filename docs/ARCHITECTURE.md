# Архитектура StudoBot

## Обзор системы

StudoBot — это система управления заданиями для преподавателя, построенная на микросервисной архитектуре с использованием контейнеризации Docker.

## Компоненты системы

### 1. Telegram Bot (Frontend)

**Роль**: Единая точка входа для студентов и преподавателя.

**Функции**:
- Приём файлов и сообщений
- Регистрация студентов при первом обращении
- Команды навигации (/start, /status, /deadlines)
- Отправка уведомлений и напоминаний

**Технологии**: python-telegram-bot или aiogram

### 2. API Server (Backend)

**Роль**: Центральный REST API для всех операций.

**Функции**:
- CRUD операции со студентами, работами, файлами
- Авторизация через Telegram ID
- AI-анализ загруженных работ
- Генерация отчётов

**Технологии**: FastAPI, SQLAlchemy, Pydantic

**Endpoints**:
```
/api/students      - Управление студентами
/api/submissions   - Управление работами
/api/files         - Загрузка и скачивание файлов
/api/reviews       - Проверки и рецензии
/api/notifications - Уведомления и рассылки
/api/webhooks      - Приём webhook'ов
```

### 3. PostgreSQL (Database)

**Роль**: Хранение метаданных и связей.

**Сущности**:
- `students` — студенты и аспиранты
- `assignment_types` — типы заданий с пайплайнами
- `submissions` — конкретные работы студентов
- `files` — версионированные файлы
- `reviews` — проверки и рецензии
- `notifications` — уведомления

**Особенности**:
- UUID как первичные ключи
- JSONB для гибких данных
- Автоматические триггеры updated_at
- Вычисляемые поля (actual_deadline)

### 4. MinIO (File Storage)

**Роль**: S3-совместимое хранилище файлов.

**Структура бакетов**:
```
student-files/
├── basic/           # Простые задания
├── coursework/      # Курсовые работы
│   ├── {year}-{term}/
│   │   └── {student_id}/
│   │       ├── topic/
│   │       ├── structure/
│   │       ├── draft/
│   │       └── final/
├── thesis/          # ВКР
├── articles/        # Научные статьи
└── projects/        # Проектные работы
```

**Особенности**:
- Версионирование через naming convention
- SHA256 checksum для контроля целостности
- Прямые ссылки на скачивание

### 5. n8n (Workflow Automation)

**Роль**: Оркестрация автоматических процессов.

**Workflows**:
1. **Приём файлов** — Telegram → MinIO → PostgreSQL → Confirmation
2. **Напоминания о дедлайнах** — Cron → DB → Telegram
3. **AI-анализ** — Trigger → AI API → DB → Notification
4. **Массовые рассылки** — API → Filter → Telegram
5. **Бэкап на Яндекс.Диск** — Cron → MinIO → WebDAV

### 6. Web Panel (опционально)

**Роль**: Веб-интерфейс для преподавателя.

**Функции**:
- Дашборд загрузки
- Kanban-доска работ
- Календарь дедлайнов
- Быстрая оценка
- Экспорт отчётов

**Технологии**: Next.js, React, TailwindCSS

## Потоки данных

### Загрузка файла студентом

```
Студент → Telegram Bot → n8n/API
         ↓
    Telegram API (file_id)
         ↓
    Скачивание файла
         ↓
    Вычисление SHA256
         ↓
    MinIO (хранение)
         ↓
    PostgreSQL (метаданные)
         ↓
    Подтверждение студенту
```

### Проверка работы преподавателем

```
Преподаватель → Web Panel/API
              ↓
    Выбор работы из списка
              ↓
    Скачивание файла из MinIO
              ↓
    (Опционально) AI-анализ
              ↓
    Создание review в БД
              ↓
    Обновление статуса работы
              ↓
    Уведомление студенту
```

### Напоминание о дедлайне

```
Cron (каждые 6 часов)
         ↓
    SQL: найти работы с дедлайном через 1/3 дня
         ↓
    Для каждой работы:
      - Проверить: напоминание уже отправлено?
      - Отправить сообщение в Telegram
      - Записать в notifications
         ↓
    Отчёт преподавателю
```

## Безопасность

### Авторизация

- Telegram ID как идентификатор пользователя
- JWT токены для API (веб-панель)
- Разделение прав: студент / староста / преподаватель

### Защита файлов

- SHA256 checksum — защита от подмены
- Временные ссылки на скачивание
- Логирование всех операций

### Изоляция

- Контейнеры Docker
- Отдельные сети для сервисов
- Переменные окружения для секретов

## Масштабирование

### Вертикальное

- Увеличение ресурсов сервера
- Оптимизация SQL-запросов
- Кэширование (Redis)

### Горизонтальное

- Репликация PostgreSQL
- MinIO distributed mode
- Несколько инстансов API за балансером

## Мониторинг и логирование

- Structured logging (structlog)
- Health check endpoints
- Prometheus metrics (опционально)
- Grafana dashboards (опционально)

## Резервное копирование

1. **База данных**: pg_dump ежедневно
2. **Файлы**: rsync / rclone на Яндекс.Диск
3. **Конфигурация**: Git-репозиторий
