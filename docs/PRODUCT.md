# DigitalTutor: Описание продукта

## Концепция

**DigitalTutor** — это персональная система управления учебными проектами, созданная специально для преподавателей российских вузов. Система объединяет Telegram-бота, AI-ассистента и инструменты управления дедлайнами в единую экосистему.

---

## Боли преподавателя

### 1. Потеря контекста взаимодействий

**Проблема:** Студенты пишут в разных мессенджерах (Telegram, WhatsApp, VK, почта). Истории сообщений очищаются, важная информация теряется.

**Решение DigitalTutor:**
- Единая точка входа через Telegram-бота
- Все коммуникации сохраняются в БД с таймстемпами
- История взаимодействий доступна в любой момент

---

### 2. Хаос в файлах

**Проблема:** Документы (курсовые, статьи, ВКР) разбросаны по разным каналам. Файлы теряются, версии путаются.

**Решение DigitalTutor:**
- Централизованное хранилище MinIO (S3-совместимое)
- Автоматическое версионирование файлов
- SHA256 контрольные суммы для доказательства сдачи в срок
- Резервное копирование на Яндекс.Диск

---

### 3. Забытые дедлайны

**Проблема:** Нет единого календаря дедлайнов. Студенты не получают напоминаний, преподаватель не видит загрузку.

**Решение DigitalTutor:**
- Визуализация всех дедлайнов на дашборде
- Автоматические напоминания за 1, 3, 7 дней
- Предупреждение о "пиках" загрузки (50+ работ одновременно)

---

### 4. Повторяющиеся ответы

**Проблема:** Одни и те же вопросы от разных студентов требуют одинаковых ответов.

**Решение DigitalTutor:**
- FAQ-система с шаблонами ответов
- Автоматические ответы на типовые вопросы через AI

---

### 5. Нет времени на первичную проверку

**Проблема:** Первичная проверка работ "по форме" занимает много времени.

**Решение DigitalTutor:**
- AI-скрининг на три критерия:
  1. **Антиплагиат** — детекция AI-генерации и заимствований
  2. **Структура** — проверка академических требований
  3. **Оформление** — соответствие ГОСТ и требованиям журналов

---

### 6. Сложность отслеживания прогресса

**Проблема:** У каждого студента своя траектория — сложно помнить, на каком этапе каждый.

**Решение DigitalTutor:**
- Milestone-система для каждого типа работы
- Визуализация прогресса на Kanban-доске
- Таймлайн по каждому студенту

---

### 7. Аспиранты: параллельные работы

**Проблема:** Аспирант ведёт 3 статьи + диссертацию + 2 эссе — всё нужно отслеживать.

**Решение DigitalTutor:**
- Поддержка параллельных работ одного студента
- Разделение типов: статья/диссертация/эссе
- Индивидуальные дедлайны для каждого типа

---

## Типы поддерживаемых работ

### Курсовая работа
```
Выбор темы → Согласование плана → Черновик → Финальная версия
     ↓              ↓                ↓              ↓
   14 дней       21 день          45 дней        60 дней
```

### ВКР (Выпускная квалификационная работа)
```
Тема → Глава 1 → Глава 2 → Глава 3 → Полный текст → Предзащита → Финал
21д     45д       70д       90д        105д          115д        125д
```

### Научная статья
```
Выбор темы → Текст статьи → Рецензирование → Публикация
   7 дней      30 дней         45 дней         60 дней
```

Артефакт: ссылка на eLIBRARY или библиографическая ссылка по ГОСТ

### Проектная работа
```
Сценарий → Реализация → Тестирование → Релиз
  14 дней    45 дней      55 дней      60 дней
```

Артефакт: файл или ссылка на внешний ресурс

## Целевая аудитория

### Основной пользователь
- Преподаватель российского вуза
- Ведёт научную работу со студентами и аспирантами
- Работает с курсовыми, ВКР, статьями
- Не хочет вводить списки студентов вручную
- Хочет помнить контекст каждого взаимодействия

### Вторичные пользователи
- **Студенты** — сдают работы через Telegram
- **Аспиранты** — ведут несколько параллельных работ
- **Старосты** — загружают работы группы (bulk-загрузка)

---

## Ключевые сценарии использования

### Сценарий 1: Студент сдаёт курсовую
1. Студент открывает бота, отправляет `/start`
2. Видит приветствие преподавателя
3. Выбирает тип работы "Курсовая"
4. Загружает файл
5. Система автоматически:
   - Создаёт submission в БД
   - Загружает файл в MinIO
   - Вычисляет SHA256
   - Отправляет подтверждение студенту
   - Уведомляет преподавателя

### Сценарий 2: AI-проверка работы
1. Преподаватель открывает дашборд
2. Видит новые работы на проверке
3. Нажимает "AI-анализ"
4. Система проверяет по трём критериям:
   - Антиплагиат: оценка AI-генерации
   - Структура: наличие новизны, методов
   - Оформление: соответствие ГОСТ
5. Преподаватель видит сводку и рекомендации

### Сценарий 3: Напоминание о дедлайне
1. Cron-задача запускается каждые 6 часов
2. Находит работы с дедлайном через 1/3 дня
3. Отправляет уведомления студентам
4. Логирует отправку в БД

## Интеграции

### Яндекс.Диск
- Резервное копирование файлов
- Доступ старостам к общим папкам
- Синхронизация с локальным MinIO

### Яндекс.DataLens
- Дашборд загрузки преподавателя
- Статистика по типам работ
- Визуализация дедлайнов

### n8n (Coolify)
- Оркестрация workflows
- Интеграция с AI-сервисами
- Автоматические напоминания

### Ollama
- Локальная AI-модель (Gemma 3 4B)
- Антиплагиат-детекция
- Проверка структуры и оформления

## Ограничения системы

- **Нет интеграции** с зачётками и балльно-рейтинговыми системами вузов

- **Работает автономно** — экспорт оценок ручной
